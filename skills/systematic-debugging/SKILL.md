# 체계적 디버깅

## 개요

무작위 수정은 시간을 낭비하고 새 버그를 만든다. 급한 패치는 근본 문제를 가린다.

**핵심 원칙:** 수정을 시도하기 전에 반드시 근본 원인을 먼저 찾는다. 증상만 고치는 것은 실패다.

**이 프로세스의 문구를 어기는 것은 디버깅의 정신을 어기는 것이다.**

## 철칙 (The Iron Law)

```
근본 원인 조사 없이는 수정도 없다
```

Phase 1을 완료하지 않았다면 수정을 제안할 수 없다.

## 언제 사용하는가

모든 기술적 문제에 사용한다:
- 테스트 실패
- 프로덕션 버그
- 예기치 않은 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 이럴 때 반드시 사용한다:**
- 시간 압박이 있을 때 (긴급할수록 찍기가 유혹적이다)
- "빠른 수정 하나"가 뻔해 보일 때
- 이미 여러 수정을 시도했을 때
- 이전 수정이 통하지 않았을 때
- 문제를 완전히 이해하지 못했을 때

**이럴 때도 건너뛰지 않는다:**
- 문제가 단순해 보일 때 (단순한 버그에도 근본 원인이 있다)
- 급할 때 (서두르면 재작업이 보장된다)
- 당장 고치라는 압박이 있을 때 (체계적인 쪽이 갈팡질팡보다 빠르다)

## 네 단계 (The Four Phases)

각 단계를 완료해야 다음 단계로 넘어갈 수 있다.

### Phase 1: 근본 원인 조사

**어떤 수정이든 시도하기 전에:**

1. **에러 메시지를 주의 깊게 읽는다**
   - 에러와 경고를 대충 넘기지 않는다
   - 정확한 해법이 담겨 있는 경우가 많다
   - 스택 트레이스를 끝까지 읽는다
   - 라인 번호, 파일 경로, 에러 코드를 기록한다

2. **일관되게 재현한다**
   - 확실하게 트리거할 수 있는가?
   - 정확한 재현 절차는?
   - 매번 발생하는가?
   - 재현이 안 되면 → 데이터를 더 모은다. 추측하지 않는다

3. **최근 변경을 확인한다**
   - 이 문제를 일으킬 만한 변경이 무엇인가?
   - git diff, 최근 커밋
   - 새 의존성, 설정 변경
   - 환경 차이

4. **다중 컴포넌트 시스템에서는 증거를 수집한다**

   **시스템이 여러 컴포넌트로 구성될 때 (CI → 빌드 → 서명, API → 서비스 → 데이터베이스):**

   **수정을 제안하기 전에 진단용 계측을 추가한다:**
   ```
   각 컴포넌트 경계마다:
     - 컴포넌트로 들어가는 데이터를 로깅
     - 컴포넌트에서 나오는 데이터를 로깅
     - 환경/설정 전파를 확인
     - 각 레이어의 상태를 확인

   한 번 실행해 어디서 깨지는지 보여주는 증거를 수집한다
   그 다음 증거를 분석해 실패한 컴포넌트를 특정한다
   그 다음 그 컴포넌트를 조사한다
   ```

   **예시 (다층 시스템):**
   ```bash
   # Layer 1: 워크플로우
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: 빌드 스크립트
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: 서명 스크립트
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: 실제 서명
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **이것이 밝혀주는 것:** 어느 레이어가 실패하는지 (secrets → workflow ✓, workflow → build ✗)

5. **데이터 흐름을 추적한다**

   **에러가 콜 스택 깊은 곳에 있을 때:**

   이 디렉토리의 `root-cause-tracing.md`에 역방향 추적 기법 전체가 있다.

   **요약 버전:**
   - 나쁜 값이 어디서 생겨나는가?
   - 무엇이 나쁜 값으로 이것을 호출했는가?
   - 원천을 찾을 때까지 계속 거슬러 올라간다
   - 증상이 아니라 원천에서 고친다

### Phase 2: 패턴 분석

**고치기 전에 패턴을 찾는다:**

1. **동작하는 예시를 찾는다**
   - 같은 코드베이스에서 비슷하게 동작하는 코드를 찾는다
   - 깨진 것과 비슷하면서 잘 동작하는 것은 무엇인가?

2. **레퍼런스와 비교한다**
   - 패턴을 구현 중이라면 레퍼런스 구현을 완독한다
   - 훑지 말고 한 줄 한 줄 읽는다
   - 패턴을 완전히 이해한 뒤 적용한다

3. **차이를 식별한다**
   - 동작하는 것과 깨진 것의 차이는 무엇인가?
   - 아무리 사소해도 모든 차이를 나열한다
   - "그건 상관없을 것"이라고 가정하지 않는다

4. **의존성을 이해한다**
   - 이것이 필요로 하는 다른 컴포넌트는?
   - 어떤 설정, 구성, 환경이 필요한가?
   - 어떤 가정을 하고 있는가?

### Phase 3: 가설과 검증

**과학적 방법:**

1. **단일 가설을 세운다**
   - 명확하게 서술한다: "X가 근본 원인이라고 생각한다. 근거는 Y다"
   - 적어둔다
   - 막연하지 않게, 구체적으로

2. **최소한으로 검증한다**
   - 가설 검증에 필요한 가장 작은 변경만 한다
   - 변수는 한 번에 하나
   - 여러 가지를 동시에 고치지 않는다

3. **확인한 뒤에 진행한다**
   - 통했는가? → Phase 4로
   - 안 통했는가? → 새 가설을 세운다
   - 실패한 수정 위에 수정을 쌓지 않는다

4. **모를 때는**
   - "X를 이해하지 못했다"라고 말한다
   - 아는 척하지 않는다
   - 도움을 요청한다
   - 더 조사한다

### Phase 4: 구현

**증상이 아니라 근본 원인을 고친다:**

1. **실패하는 테스트 케이스를 만든다**
   - 가능한 가장 단순한 재현
   - 가능하면 자동화된 테스트
   - 프레임워크가 없으면 일회성 테스트 스크립트
   - 고치기 전에 반드시 있어야 한다
   - 수정 전에 실패하는 테스트로 재현한다 — 계약을 먼저 정하고, 테스트가 그 계약을 인코딩하게 한다

2. **단일 수정을 구현한다**
   - 식별한 근본 원인을 해결한다
   - 변경은 한 번에 하나
   - "온 김에" 개선 금지
   - 리팩토링 끼워넣기 금지

3. **수정을 검증한다**
   - 테스트가 이제 통과하는가?
   - 다른 테스트가 깨지지 않았는가?
   - 문제가 실제로 해결됐는가?
   - 프로젝트의 테스트·타입체크·린트를 실행해 검증한 뒤에 완료를 선언한다

4. **수정이 통하지 않으면**
   - 멈춘다
   - 센다: 지금까지 수정을 몇 번 시도했는가?
   - 3회 미만: Phase 1로 돌아가 새 정보로 재분석한다
   - **3회 이상: 멈추고 아키텍처를 의심한다 (아래 5번)**
   - 아키텍처 논의 없이 4번째 수정을 시도하지 않는다

5. **수정 3회 이상 실패: 아키텍처를 의심한다**

   **아키텍처 문제를 가리키는 패턴:**
   - 수정할 때마다 다른 곳에서 새로운 공유 상태/결합/문제가 드러난다
   - 수정에 "대규모 리팩토링"이 필요하다
   - 수정할 때마다 다른 곳에서 새 증상이 생긴다

   **멈추고 근본을 의심한다:**
   - 이 패턴이 근본적으로 건전한가?
   - "관성으로 붙들고 있는" 것은 아닌가?
   - 증상 수정을 계속하는 대신 아키텍처를 리팩토링해야 하는가?

   **더 수정을 시도하기 전에 사용자와 논의한다**

   이것은 실패한 가설이 아니라 잘못된 아키텍처다.

## 위험 신호 — 멈추고 프로세스로 돌아가라

이런 생각이 들면:
- "일단 빠르게 고치고 나중에 조사하자"
- "그냥 X를 바꿔보고 되는지 보자"
- "여러 변경을 넣고 테스트 돌리자"
- "테스트는 생략, 수동으로 확인하지"
- "아마 X일 거야, 그걸 고치자"
- "완전히 이해는 못 했지만 이러면 될 것 같다"
- "패턴은 X라는데 다르게 응용하겠다"
- "주요 문제는 이것들이다: [조사 없이 수정 나열]"
- 데이터 흐름을 추적하기 전에 해법부터 제안
- **"한 번만 더 고쳐보자" (이미 2회 이상 시도했을 때)**
- **수정할 때마다 다른 곳에서 새 문제가 드러날 때**

**전부 의미하는 바는 하나: 멈추고 Phase 1로 돌아가라.**

**수정 3회 이상 실패했다면:** 아키텍처를 의심하라 (Phase 4의 5번)

## 잘못 가고 있다는 사용자의 신호

**이런 지적을 주시한다:**
- "그게 안 되고 있는 건가요?" — 확인 없이 가정했다
- "그걸 보여주나요?" — 증거 수집을 먼저 했어야 했다
- "추측 그만하세요" — 이해 없이 수정을 제안하고 있다
- "더 깊게 생각해보세요" — 증상이 아니라 근본을 의심하라
- "막힌 건가요?" (답답함) — 접근이 통하지 않고 있다

**이 신호가 보이면: 멈추고 Phase 1로 돌아간다.**

## 흔한 합리화

| 변명 | 현실 |
|------|------|
| "단순한 문제라 프로세스가 필요 없다" | 단순한 문제에도 근본 원인이 있다. 단순한 버그일수록 프로세스도 빠르다. |
| "긴급 상황이라 프로세스 돌릴 시간이 없다" | 체계적 디버깅이 찍고-확인하는 갈팡질팡보다 빠르다. |
| "일단 이것부터 해보고 나중에 조사하자" | 첫 수정이 패턴을 만든다. 처음부터 제대로 한다. |
| "수정이 통하는지 확인한 뒤 테스트를 쓰겠다" | 테스트 없는 수정은 오래가지 않는다. 테스트 먼저 쓰는 것이 증명이다. |
| "여러 수정을 한꺼번에 하면 시간이 절약된다" | 뭐가 통했는지 분리할 수 없다. 새 버그를 만든다. |
| "레퍼런스가 너무 길어서 패턴만 응용하겠다" | 부분적 이해는 버그를 보장한다. 완독하라. |
| "문제가 보인다, 바로 고치자" | 증상이 보이는 것 ≠ 근본 원인 이해. |
| "한 번만 더" (2회 이상 실패 후) | 3회 이상 실패 = 아키텍처 문제. 또 고치지 말고 패턴을 의심하라. |

## 빠른 참조

| 단계 | 핵심 활동 | 성공 기준 |
|------|----------|----------|
| **1. 근본 원인** | 에러 읽기, 재현, 변경 확인, 증거 수집 | 무엇이·왜인지 이해 |
| **2. 패턴** | 동작 예시 찾기, 비교 | 차이 식별 |
| **3. 가설** | 이론 수립, 최소 검증 | 확증 또는 새 가설 |
| **4. 구현** | 테스트 작성, 수정, 검증 | 버그 해결, 테스트 통과 |

## 프로세스가 "근본 원인 없음"을 밝혀낼 때

체계적 조사 결과 문제가 정말로 환경적, 타이밍 의존적, 외부 요인이라면:

1. 프로세스를 완주한 것이다
2. 무엇을 조사했는지 문서화한다
3. 적절한 처리를 구현한다 (재시도, 타임아웃, 에러 메시지)
4. 향후 조사를 위한 모니터링/로깅을 추가한다

**단:** "근본 원인 없음" 사례의 95%는 조사가 불완전한 것이다.

## 지원 기법

체계적 디버깅의 일부로, 이 디렉토리에서 볼 수 있다:

- **`root-cause-tracing.md`** — 콜 스택을 거슬러 올라가 최초 트리거를 찾는 역방향 추적
- **`defense-in-depth.md`** — 근본 원인을 찾은 뒤 여러 레이어에 검증 추가
- **`condition-based-waiting.md`** — 임의의 타임아웃을 조건 폴링으로 대체

## 실측 효과

디버깅 세션 데이터:
- 체계적 접근: 수정까지 15-30분
- 무작위 수정 접근: 2-3시간 갈팡질팡
- 첫 수정 성공률: 95% vs 40%
- 새로 유입된 버그: 거의 0 vs 흔함
