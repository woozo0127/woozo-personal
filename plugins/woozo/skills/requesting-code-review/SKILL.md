# 코드 리뷰 요청

코드 리뷰어 서브에이전트를 디스패치해 문제가 번지기 전에 잡는다. 리뷰어에게는 세션 히스토리가 아니라 정밀하게 다듬은 컨텍스트만 전달한다 — 리뷰어는 사고 과정이 아닌 작업 결과물에 집중하고, 메인 세션의 컨텍스트는 이어지는 작업을 위해 보존된다.

**핵심 원칙:** 일찍 리뷰하고, 자주 리뷰한다.

## 언제 리뷰를 요청하는가

**필수:**
- 구현 계획의 태스크를 완료할 때마다
- 주요 기능을 완성했을 때
- main에 머지하기 전

**선택이지만 가치 있음:**
- 막혔을 때 (새로운 시각)
- 리팩토링 전 (베이스라인 확인)
- 복잡한 버그를 수정한 후

## 요청 방법

**1. git SHA 범위를 확보한다:**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 또는 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. 코드 리뷰어 서브에이전트를 디스패치한다:**

`general-purpose` 서브에이전트에 이 디렉토리의 `code-reviewer.md` 템플릿을 채워 디스패치한다.

**플레이스홀더:**
- `[DESCRIPTION]` — 무엇을 만들었는지 간단한 요약
- `[PLAN_OR_REQUIREMENTS]` — 무엇을 해야 하는지 (계획 파일 경로, 태스크 내용, 또는 요구사항)
- `[BASE_SHA]` — 시작 커밋
- `[HEAD_SHA]` — 끝 커밋

**3. 피드백을 반영한다:**
- Critical 이슈는 즉시 수정한다
- Important 이슈는 다음 단계로 넘어가기 전에 수정한다
- Minor 이슈는 나중을 위해 기록한다
- 리뷰어가 틀렸으면 기술적 근거를 들어 반박한다

## 예시

```
[태스크 2 완료: 검증 함수 추가]

나: 다음으로 넘어가기 전에 코드 리뷰를 요청하겠습니다.

BASE_SHA=$(git log --oneline | grep "Task 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[코드 리뷰어 서브에이전트 디스패치]
  DESCRIPTION: verifyIndex()와 repairIndex() 추가, 이슈 유형 4종 처리
  PLAN_OR_REQUIREMENTS: docs/plans/deployment-plan.md의 태스크 2
  BASE_SHA: a7981ec
  HEAD_SHA: 3df7661

[서브에이전트 응답]:
  강점: 깔끔한 아키텍처, 실질적인 테스트
  이슈:
    Important: 진행률 표시 누락
    Minor: 리포팅 주기의 매직 넘버(100)
  판정: 진행 가능

나: [진행률 표시 수정]
[태스크 3으로 진행]
```

## 워크플로우 통합

**구현 계획 실행 중:**
- 태스크 완료마다, 또는 자연스러운 체크포인트마다 리뷰한다
- 문제가 쌓이기 전에 잡고, 수정한 뒤 다음 태스크로 넘어간다

**일반 개발:**
- 머지 전에 리뷰한다
- 막혔을 때 리뷰한다

## 위험 신호

**절대 하지 않는다:**
- "단순하니까" 리뷰 생략
- Critical 이슈 무시
- Important 이슈를 수정하지 않은 채 진행
- 유효한 기술적 피드백에 반박을 위한 반박

**리뷰어가 틀렸다면:**
- 기술적 근거를 들어 반박한다
- 동작을 증명하는 코드/테스트를 제시한다
- 불명확하면 해명을 요청한다

템플릿: 이 디렉토리의 `code-reviewer.md`
